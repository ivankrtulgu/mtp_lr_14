**Инструмент:** Sisyphus (Gemma 4 31B)
**Промпт:**
```
Ты — системный архитектор. Разработай структуру проекта для ETL-конвейера (Go -> Python -> SQL).
Проект должен поддерживать:
1. Разделение данных по стадиям: raw (исходные), intermediate (промежуточные), processed (очищенные).
2. Изоляция компонентов: отдельная директория для Go-сборщика и для Python-аналитики.
3. Хранение артефактов: отдельная папка для визуализаций и документации.
4. Стандартизация путей: относительные пути между компонентами для обеспечения переносимости.

Результат: схема дерева каталогов с описанием назначения каждой папки.
```

**Результат:** Спроектирована и внедрена иерархическая структура проекта:
- `/data/raw` — исходные CSV-файлы.
- `/data/intermediate` — JSONL-вывод сборщика.
- `/data/processed` — итоговые Parquet-файлы.
- `/collector` — исходный код на Go (сборщик).
- `/analysis` — исходный код на Python (ETL, анализ, SQL).
- `/plots` — графики и визуализации.
- `/docs` — техническая документация.

Данная структура обеспечивает прозрачный жизненный цикл данных (Data Lineage) и предотвращает смешивание кода разных языков.

## Задание 2: Добавление буферизации и пакетной записи

### Промпт 1
**Инструмент:** Sisyphus $\to$ golang-pro (Gemma 4 31B)
**Промпт:**
```
Ты — senior Go разработчик. Модифицируй существующий сборщик данных на Go для оптимизации операций ввода-вывода (I/O).

Требования:
1. Реализовать пакетную запись (Batch Writing) вместо записи каждой записи по отдельности.
2. Внедрить механизм буферизации:
   - Параметр N (BatchSize = 100): данные должны сбрасываться на диск, как только в буфере накопится 100 записей.
   - Параметр T (BatchTimeout = 2s): данные должны сбрасываться на диск каждые 2 секунды, даже если буфер не заполнен до N.
3. Обеспечить корректный финальный сброс (flush) всех оставшихся в буфере данных при закрытии канала результатов.
4. Сохранить существующую архитектуру с Worker Pool и горутинами.

Результат: обновленный файл main.go с реализованной логикой буферизации и таймером.
```

**Результат:** Модифицирован сборщик данных. Внедрена стратегия Timed-Batching с использованием `time.Ticker` и внутреннего слайса-буфера. Это позволило сократить количество системных вызовов записи в файл, объединяя по 100 транзакций в одну операцию записи. Верификация через логи подтвердила корректную работу пакетного сброса.

### Итого
- **Количество промптов:** 1
- **Что пришлось исправлять вручную:** Провел ручную верификацию работы таймера, временно увеличив BatchSize, чтобы убедиться, что сброс данных происходит по истечении T секунд даже при незаполненном буфере.
- **Время:** ~10 мин
